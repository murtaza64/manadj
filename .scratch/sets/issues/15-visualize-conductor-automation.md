# 15 — Visualize Conductor automation on the performance controls

(Renumbered from 13, 2026-07-05 — parallel filing collided with 13-rehearsal-loop.)

Status: ready-for-agent

## Parent

.scratch/sets/PRD.md

## Problem

During Conductor playback the on-screen mixer faders/EQ/filter (and MIDI
LED feedback) sit still: the Conductor writes through the Mixer's
automation overlay, which never touches base state or notifies subscribers
(ADR 0022 — deliberate, so user knob positions survive and the capture
recorder sees base state only). The Performance view therefore shows a
mix that doesn't match what's sounding. Same gap applies to the editor's
MixPlayer. Noticed during the 04 review click-through (2026-07-05).

## Decisions (triaged 2026-07-05)

- **Screen-only, v1.** MIDI LED Feedback keeps mirroring base state — LED
  rings can't show two positions, and automation-driven LEDs would muddy
  the takeover intuition (the physical knob's position IS base state).
  Deferred, not rejected.
- **Ghost = second indicator, never a moving control.** The real thumb
  stays at base (the hand's position, the takeover target); a translucent
  DAW-style marker shows the live automation value. Controls never animate
  themselves. Model the visual on DAW automation-read affordances; expect
  visual iteration with the human after the first cut.

## Comments

> *This was generated by AI during triage.*

Verified against the code: the Mixer's automation overlay owns the 10
lane-driven params and neither mutates base state nor notifies subscribers,
so control surfaces (which render base state) sit still while automation
sounds. No automation read API or ghost indicator exists anywhere today;
the editor's timeline lanes are the only automation visualization. Not
already implemented; no prior rejection on record.

## Agent Brief

**Category:** enhancement
**Summary:** Read-only ghost indicators on the performance mixer controls showing live automation values while an automation overlay is engaged

**Current behavior:**
While the Conductor (or the Transition editor's mix player) is sounding,
the Mixer's automation overlay drives faders/EQ/filter, but the on-screen
controls render base state only — the visible mixer doesn't match the
audible mix. Nothing exposes the overlay's current values for display.

**Desired behavior:**
Whenever an automation overlay is engaged, each automated control on the
performance mixer strips shows a translucent DAW-style ghost marker at the
live automation value, updated per-frame (rAF, like the waveform
playheads), alongside the real thumb which stays at base state. Ghosts
appear only while the overlay is engaged and only on channels/params the
overlay currently owns; they vanish on disengage and at takeover. The
base/overlay split is untouched: user gestures, capture, and persistence
keep reading and writing base state exclusively. Works identically for
Conductor playback and editor auditions (both engage the same overlay).

**Key interfaces:**
- A small read-only accessor on the Mixer exposing the engaged overlay's
  current per-channel values (e.g. `getAutomation(channel)` returning the
  automated param values or null when disengaged) — no subscriber
  notifications, polled per-frame by the view
- The mixer strip control components — accept/render an optional ghost
  value; no change to their gesture handling or value propagation
- No changes to the automation write path, capture recorder, or MIDI
  Feedback (Feedback continues to mirror base state)

**Acceptance criteria:**
- [ ] During Conductor playback, fader/EQ/filter ghosts track the audible
      automation; real thumbs stay at base positions
- [ ] Same during Transition-editor auditions
- [ ] Ghosts disappear on overlay disengage and at takeover, instantly
- [ ] Dragging a control mid-automation still writes base state and
      triggers takeover exactly as before (no behavior change)
- [ ] No Takes are captured differently (capture reads base state only)
- [ ] MIDI LED feedback unchanged

**Out of scope:**
- MIDI LED/Feedback mirroring of automation values (deferred by decision)
- Crossfader/master or any param the overlay doesn't own
- Visual polish beyond a first DAW-style cut — expect an iteration round
  with the human on the ghost styling
- Any change to automation semantics, base/overlay ownership, or ADR 0022
