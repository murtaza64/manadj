# 18 — Set view: row selection + group drag-reorder

Status: ready-for-agent

## Parent

.scratch/sets/PRD.md

## What to build

The Set detail view has no row selection: `SetTrackRow` always drags
`[trackId]` alone. Dragging a *group* of tracks to reorder a set is a
natural act (split out of the 17 grill, 2026-07-05; triaged to
ready-for-agent 2026-07-05 — interaction decisions below are settled).

Half the machinery exists: the drag payload format carries id arrays,
and the Set pane's reorder drop path already accepts plural dropped ids.
What's missing is the selection model and its interaction design.

## Decisions (triaged 2026-07-05)

- **Selection gestures, standard**: click = select one; shift-click =
  range; cmd-click = toggle. Dragging a selected row drags the whole
  selection; dragging an unselected row selects-and-drags just it.
  Esc or empty-space click clears. Selection visuals coexist with the
  conducting-row highlight (distinct colors, both visible at once).
- **Non-contiguous selections compact on drop**: the selected rows land
  as one contiguous run, in set order, at the drop point (playlist-editor
  convention). Pins interior to each previously-contiguous sub-run
  survive the move; every other affected edge degrades per the pin rules.
- **Delete/Backspace removes the selection from the Set** — identical
  pin handling to today's single-row removal.

## Agent Brief

**Category:** enhancement
**Summary:** Row selection in the Set detail view with group drag-reorder, selection-aware context menu, and keyboard removal

**Current behavior:**
Set rows have no selection; drag-reorder moves one track; the context
menu (17) targets the clicked row; removal is the hover ✕ per row.

**Desired behavior:**
Per the Decisions above. Group drag shows the existing live drag preview
with the group's pin consequences (edge adjacencies breaking/restoring);
internal pins of a contiguous block survive a move. Issue 17's menu acts
on the selection automatically (its targeting rule prefers the selection;
its menu core is multi-aware) — verify, don't rebuild. Edge-pin
semantics follow issue 07's dormancy rules if 07 has landed; plain
degradation otherwise (coordinate — 07 is expected to land first).

**Key interfaces:**
- A selection model for the Set pane (ids + anchor for range), living
  beside the Set view's store — not component-local state, so the menu
  and keyboard handlers can read it
- The drag payload already carries id arrays; the reorder application
  already accepts plural dropped ids — group drag should converge on
  those, adding compaction ordering
- The live drag-preview/plan recompute must accept the hypothetical
  multi-row order

**Acceptance criteria:**
- [ ] Single/range/toggle selection work without breaking drag-reorder,
      hover ▶/✕, or the conducting highlight
- [ ] Dragging a selected row moves the whole selection; a contiguous
      block keeps its internal pins
- [ ] A non-contiguous selection compacts at the drop point in set order;
      only the true edges degrade, and the drag preview shows it
- [ ] Right-click with a selection: 17's menu acts on all selected rows
      (multi labels, Load disabled on multi)
- [ ] Delete/Backspace removes the selected rows with standard pin
      handling
- [ ] Esc / empty-space click clears; selection and conducting highlight
      render simultaneously and distinctly

**Out of scope:**
- Selection in the Library table (exists already) or any other surface
- Cross-surface drag of multi-selections into the Set (separate concern)
- Any change to 17's menu core or 07's dormancy rules — consume, don't
  modify
