# Handoff: set-OPEN lag (post-42 residual) — for the performance lane

## Symptom (human report, 2026-07-06, after 42's fix landed)

Opening a Set still makes the UI unresponsive for a few seconds (worst on
first/initial load). Issue 42 fixed **interaction** jank (row clicks: 176
row re-renders → 2-4, interact.js green) — but its loop measures clicks on
an ALREADY-OPEN set. Set **open** was never measured. Treat this as a new
phase: extend the harness with an open-set probe before touching anything.

## Diagnosis so far (this session, sets 30/ladder lane — file:line from a
subagent pass over the open path; verify lines against current main)

Cleared (measured or verified):
- Ladder clip rasterization: frame-budgeted queue, 6ms/frame
  (`OverviewLadder.tsx` scheduleClipDraw, change `mkkxptqx`) — mount fill
  is progressive, ~4ms/clip jobs.
- Blob decode: ~1.5ms/track (measured on real blobs) — though note it
  runs per CONSUMER per mount (`useWaveformBlob.ts:24-27` useMemo, only
  the ArrayBuffer is query-cached) — 88 clips ≈ 130ms, re-paid every set
  switch. Cheap-ish but a free win: decode inside queryFn instead.
- No audio decode on set open (prefetchTrackBuffer is Conductor-verb-only).

Open suspects, in ranked order:

1. **Query-resolution render burst during mount.** Set open fires ~25-60
   staggered resolutions for 18 tracks (hundreds for 88): per-track
   hot-cue queries at `staleTime: 0` mounted THREE times (SetDetailPane
   ~:330 + two useSetPlan instances — plan and previewPlan, useSetPlan
   ~:103) → up to 3 waves of N GETs, every resolution notifying all three
   observer sets and re-rendering the pane. 42's memo() made each
   re-render cheap, but parent-level work (memo dep comparisons, derived
   maps, ladder parent) still runs per resolution, and in dev StrictMode
   it doubles. For 88 tracks this is plausibly still seconds of chopped
   main thread. Candidate fixes: staleTime > 0 on hot cues, dedupe the
   triple observer set (single shared subscription), or a bulk hot-cues
   endpoint (N queries → 1).

2. **One big late commit.** The plan stays undefined until the LAST
   hot-cue/take query resolves (useSetPlan ~:122), so the ladder (~90
   absolute divs + N canvases) and all plan-dependent cells mount in one
   large commit right at the burst tail — likely the single worst frame.

3. **Request count itself**: 88-track open ≈ 88 blobs (~18MB) + 264
   hot-cue GETs + 88 track GETs through the vite proxy to the dev
   backend. Even cheap handlers × 400+ resolutions = sustained main-thread
   chop. Bulk endpoints or query batching would collapse this.
